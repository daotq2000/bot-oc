## Problem
Bot was making hundreds of requests per second to /fapi/v1/klines, causing:
- Rate limit 429 errors
- IP ban: "Way too many requests; IP banned until..."
- Market data request failures across 541 symbols with 1082 streams

Root cause: 22 WebSocket connections all FAILED (connected=false), causing
aggressive REST API fallback on every price tick.

## Solution

### 1. RealtimeOCDetector.js
- DISABLED REST API fallback for candle opens (main cause of rate limit)
- Added config: OC_REST_FALLBACK_ENABLED=false (default)
- Kept code commented for reference with proper safeguards if needed
- Rely on WebSocket data + prev_close fallback only

### 2. BinanceDirectClient.js
- Integrated binanceRequestScheduler for all market data requests
- Added circuit breaker checks before every request
- Added rate limit blocking checks (10s cooldown on 429)
- Increased interval: 200ms ‚Üí 500ms for market data
- Created _makeMarketDataRequestInternal() for proper request handling
- All requests now go through centralized protection layers:
  * Rate limit block check
  * Circuit breaker check  
  * Request scheduler (8 req/sec limit)
  * Exponential backoff on 429
  * Longer timeout for market data (20s)

### 3. IndicatorWarmup.js
- Switched from raw fetch() to BinanceDirectClient
- Inherit all protections: scheduler, circuit breaker, rate limiting
- Added config: INDICATOR_WARMUP_USE_LEGACY_FETCH=false (default)
- Proper 429 handling with exponential backoff

### 4. Documentation
- Created docs/RATE_LIMIT_FIX_CONFIG.md with 86 config parameters
- Created RATE_LIMIT_FIX_SUMMARY.md with deployment guide
- Created scripts/test-rate-limit-fix.sh for verification

## Configuration Changes (Defaults)

New configs (all with safe defaults):
- BINANCE_USE_SCHEDULER_FOR_MARKET_DATA=true
- BINANCE_MARKET_DATA_MIN_INTERVAL_MS=500 (increased from 200)
